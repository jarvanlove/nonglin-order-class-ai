# 06 - 开发流程规范

> 目标：定义从需求到上线的完整流程，让学生知道每一步该做什么、用什么工具、产出什么。

---

## 总体流程图

```
需求描述 → AI生成代码 → 人工审查 → 运行测试 → 集成联调 → 部署上线
   ↑___________________________________________________________↓
                        （循环迭代）
```

---

## Phase 1：需求描述（学生做）

**目标**：把"我想做什么"转化为 AI 能理解的"结构化需求"

**步骤**：
1. 用一句话描述功能（如："我要做一个农产品列表页面，能显示图片、名称、价格"）
2. 补充约束条件（如："要响应式布局，手机也能看"）
3. 补充数据要求（如："数据从 /api/products 获取"）

**产出**：一段完整的提示词（Prompt）

---

## Phase 2：AI 生成代码（Claude Code CLI 做）

**目标**：让 AI 根据需求生成可用代码

**步骤**：
1. 打开终端，进入项目目录
2. 运行 `claude`
3. 粘贴提示词
4. AI 生成代码后，审查生成的内容

**审查清单**：
- [ ] 代码是否符合编码规范（命名、结构）
- [ ] 是否有明显的逻辑错误
- [ ] 是否引入了不需要的依赖
- [ ] 是否处理了错误情况

**产出**：新增/修改的代码文件

---

## Phase 3：人工审查与调整（学生做）

**目标**：确保 AI 生成的代码符合项目需求

**步骤**：
1. 阅读 AI 生成的代码，理解其逻辑
2. 检查边界情况（空数据、错误输入等）
3. 调整不符合规范的命名或结构
4. 补充 AI 遗漏的错误处理

**常用 Claude Code CLI 命令**：
```bash
# 让 AI 解释某段代码
/explain src/components/ProductList.tsx

# 让 AI 优化某段代码
/refactor src/utils/format.ts

# 让 AI 添加测试
/test src/components/ProductCard.tsx

# 让 AI 修复报错
/fix
```

**产出**：审查后的代码 + 修改记录

## Phase 4：运行测试（学生 + AI 协作）

### 前端测试

1. **启动开发服务器**：
   ```bash
   cd frontend
   pnpm install
   pnpm dev
   ```

2. **浏览器验证**：
   - 打开 `http://localhost:5173`
   - 检查页面是否正常渲染
   - 检查交互是否正常（点击、输入、提交）
   - 打开浏览器开发者工具（F12），查看 Console 是否有报错

3. **响应式测试**：
   - 按 F12 打开开发者工具
   - 点击设备模拟按钮（手机图标）
   - 测试不同屏幕尺寸下的显示效果

### 后端测试

1. **启动后端服务**：
   ```bash
   cd backend
   python -m venv venv
   venv\Scripts\activate  # Windows
   pip install -r requirements.txt
   uvicorn main:app --reload
   ```

2. **API 测试**：
   - 打开 `http://localhost:8000/docs`（FastAPI 自动生成的 Swagger UI）
   - 直接在线测试各个接口
   - 或使用 Postman / curl 测试

3. **数据库验证**：
   - 打开 pgAdmin
   - 检查数据是否正确写入

## Phase 5：前后端联调（学生做）

**目标**：确保前端能正确调用后端 API

**步骤**：
1. 同时启动前端（`pnpm dev`）和后端（`uvicorn main:app --reload`）
2. 在前端页面触发 API 请求
3. 打开浏览器开发者工具 → Network 标签
4. 检查请求是否发出、响应是否正确

**常见问题排查**：

| 现象 | 可能原因 | 解决方案 |
|------|---------|---------|
| 前端报错 "Network Error" | 后端没启动或端口不对 | 检查后端是否运行在 8000 端口 |
| 后端返回 404 | URL 路径错误 | 检查前端请求的 URL 和后端定义的路由是否一致 |
| 后端返回 422 | 请求参数格式不对 | 检查请求体是否符合 Pydantic 模型定义 |
| 后端返回 500 | 后端代码报错 | 查看后端终端的报错信息，用 `/fix` 让 AI 修复 |
| CORS 错误 | 跨域配置问题 | 检查后端是否配置了 CORS |

## Phase 6：Git 版本管理

### 提交规范

使用约定式提交（Conventional Commits）：

```
<type>(<scope>): <subject>

<body>
```

常用类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(product): 添加产品列表页面` |
| `fix` | 修复 Bug | `fix(api): 修复价格计算错误` |
| `docs` | 文档更新 | `docs(readme): 更新安装说明` |
| `style` | 代码格式 | `style(components): 统一缩进` |
| `refactor` | 重构 | `refactor(utils): 优化日期格式化函数` |

### 分支策略（简化版）

```
main 分支（稳定版本）
  ↑
dev 分支（日常开发）
  ↑
feature/xxx 分支（功能开发）
```

学生阶段建议：所有人直接在 `main` 分支开发，每次完成一个功能就提交一次。

## Phase 7：部署上线

### 前端部署（GitHub Pages）

1. 在 `vite.config.ts` 中设置 `base: '/my-project/'`
2. 运行 `pnpm build` 生成 `dist/` 目录
3. 推送代码到 GitHub
4. 在 GitHub 仓库设置中启用 Pages，选择 `gh-pages` 分支

### 后端部署（Render / 阿里云）

1. 准备 `requirements.txt`
2. 准备 `Procfile` 或启动脚本
3. 配置环境变量（数据库连接字符串、API Key 等）
4. 推送到代码仓库，绑定部署服务

**注意**：教学场景下，如果部署有困难，可以只在本地演示，确保项目能完整运行即可。
