# 农林高校AI工具实训课程 · 项目评分标准（终版）

> 适用于 Day 6 项目路演评分，也适用于学生自评与组间互评。

---

## 一、评分维度与权重

| 维度 | 权重 | 满分 | 核心考察点 |
|------|------|------|-----------|
| **功能完整性** | 25% | 25 分 | 需求实现度、MVP 达成情况、核心流程可演示 |
| **技术实现** | 25% | 25 分 | 工具使用深度、代码质量、三工具集成度 |
| **团队协作** | 20% | 20 分 | 分工合理性、Git 协作规范、冲突解决能力 |
| **文档质量** | 15% | 15 分 | README 完整性、代码注释、路演 PPT 清晰度 |
| **路演表现** | 15% | 15 分 | 时间控制、表达清晰度、答辩应对、演示流畅度 |

**总分**：100 分

---

## 二、各维度评分细则

### 1. 功能完整性（25 分）

| 等级 | 分值区间 | 标准描述 |
|------|----------|----------|
| 优秀 | 22-25 分 | 核心功能全部实现，至少有一个亮点功能；主流程无阻断性 Bug；有真实数据或合理模拟数据支撑 |
| 良好 | 17-21 分 | 核心功能基本实现，存在轻微 Bug 但不影响主流程演示；数据完整 |
| 合格 | 12-16 分 | 核心功能部分实现，主流程可跑通但存在明显缺陷；数据基本完整 |
| 不合格 | 0-11 分 | 核心功能缺失严重，或主流程无法演示；数据缺失或明显虚假 |

**检查清单**：

- [ ] 项目有明确的用户场景和痛点定义（至少 1 句话描述）
- [ ] 核心功能清单中的功能已实现 ≥ 60%
- [ ] 至少有一个完整的端到端流程可以现场演示
- [ ] 界面可交互（不是静态 PPT 或截图）
- [ ] 数据真实或模拟合理（不出现"test123"、"aaa"等明显测试数据）

---

### 2. 技术实现（25 分）

| 等级 | 分值区间 | 标准描述 |
|------|----------|----------|
| 优秀 | 22-25 分 | 至少使用 2 个课程工具（QClaw/WorkBuddy/Trae），且有明确的工具联动；代码结构清晰，有模块化意识；使用了 API/数据库/图表等进阶技术 |
| 良好 | 17-21 分 | 使用了 2 个课程工具，但联动较浅；代码可运行，结构基本合理；有至少一项进阶技术 |
| 合格 | 12-16 分 | 使用了 1-2 个课程工具，但仅作为独立使用；代码能运行但结构较乱；无进阶技术 |
| 不合格 | 0-11 分 | 未使用课程工具，或工具使用与项目无关；代码无法运行或存在严重错误 |

**检查清单**：

- [ ] 使用了 QClaw（信息收集/自动化/插件）
- [ ] 使用了 WorkBuddy（文件处理/批量操作/报表生成）
- [ ] 使用了 Trae（代码生成/项目开发/调试）
- [ ] 工具间有数据流转（如 QClaw 收集 → WorkBuddy 处理 → Trae 展示）
- [ ] 代码有基本的分层/模块化（HTML/CSS/JS 分离，或前后端分离）
- [ ] 使用了至少一项进阶技术：外部 API、数据库（SQLite）、数据可视化（ECharts）、定时任务等

---

### 3. 团队协作（20 分）

| 等级 | 分值区间 | 标准描述 |
|------|----------|----------|
| 优秀 | 17-20 分 | 分工明确且合理，每人有独立贡献模块；Git 提交规范（commit message 清晰、分支使用合理）；能有效解决合并冲突；每日站会有记录 |
| 良好 | 13-16 分 | 分工较明确，偶有协作摩擦；Git 使用基本规范，偶有提交混乱；能解决常见冲突 |
| 合格 | 9-12 分 | 分工存在模糊地带，部分成员贡献度偏低；Git 仅用于最终提交，无协作流程；冲突解决依赖教师/助教 |
| 不合格 | 0-8 分 | 分工严重不均，存在"搭便车"现象；未使用 Git 或仅由一人提交；团队内部冲突未解决 |

**检查清单**：

- [ ] 团队有明确的分工表（谁负责前端/后端/数据/文档/路演）
- [ ] Git 仓库有 ≥ 10 次提交，且分布于 ≥ 3 个成员
- [ ] Commit message 有意义（如"feat: 添加天气查询接口"，不是"update"、"111"）
- [ ] 无未解决的合并冲突痕迹（如代码中残留 `<<< HEAD`）
- [ ] 每个成员能在路演中讲清楚自己负责的部分

---

### 4. 文档质量（15 分）

| 等级 | 分值区间 | 标准描述 |
|------|----------|----------|
| 优秀 | 13-15 分 | README 包含：项目简介、安装步骤、功能截图、团队分工、技术栈；代码有充分注释；PPT 结构清晰、视觉统一、页数适中（5-8 页） |
| 良好 | 10-12 分 | README 基本完整，缺少 1 项次要内容；代码注释覆盖主要函数；PPT 结构合理但视觉一般 |
| 合格 | 6-9 分 | README 仅有简单描述；代码缺少注释；PPT 页数过多或过少，信息密度不均 |
| 不合格 | 0-5 分 | 无 README 或 README 与项目无关；代码零注释；PPT 缺失或纯文字堆砌 |

**检查清单**：

- [ ] README.md 存在于项目根目录
- [ ] README 包含：项目一句话简介、安装/运行步骤、功能截图或 GIF、团队成员与分工、使用的主要技术
- [ ] 核心函数/复杂逻辑有注释（不是每行都注释，而是解释"为什么"）
- [ ] 路演 PPT 控制在 5-8 页
- [ ] PPT 包含：痛点、方案、演示截图/GIF、技术栈、团队分工

---

### 5. 路演表现（15 分）

| 等级 | 分值区间 | 标准描述 |
|------|----------|----------|
| 优秀 | 13-15 分 | 时间控制精准（5±0.5 分钟）；表达流畅，逻辑清晰；演示无卡顿；答辩应对得体，能承认不足并给出改进思路 |
| 良好 | 10-12 分 | 时间基本合规（4-6 分钟）；表达较清晰，偶有口误；演示有小卡顿但不影响整体；答辩能回答大部分问题 |
| 合格 | 6-9 分 | 时间偏差较大（<4 分钟或 >7 分钟）；表达略显混乱；演示有明显卡顿或失败；答辩部分问题无法回答 |
| 不合格 | 0-5 分 | 严重超时或严重缺时；表达不清，逻辑混乱；演示主要功能失败；答辩态度消极或编造答案 |

**检查清单**：

- [ ] 路演时长控制在 5 分钟（不含评委问答）
- [ ] 开场 30 秒内讲清楚"解决什么问题"
- [ ] 有现场演示（不是只放 PPT）
- [ ] 演示流程提前预演，无重大 Bug
- [ ] 答辩时诚实回答，不编造

---

## 三、通过标准

### 课程通过条件（需同时满足）

1. **总分 ≥ 60 分**
2. **每个维度得分 ≥ 该维度满分的 40%**（即：功能 ≥ 10 分、技术 ≥ 10 分、协作 ≥ 8 分、文档 ≥ 6 分、路演 ≥ 6 分）
3. **每个团队成员必须有可辨识的个人产出**（Git 提交记录、文档署名、路演中明确分工）

### 成绩等级

| 等级 | 总分区间 | 说明 |
|------|----------|------|
| 优秀 | 90-100 分 | 可作为课程优秀项目展示，推荐参加后续比赛 |
| 良好 | 75-89 分 | 项目完整，有改进空间 |
| 合格 | 60-74 分 | 达到课程基本要求 |
| 不合格 | 0-59 分 | 需补充完善后重新提交 |

## 四、评分流程

### 评委组成

- 主讲教师（1 人）：权重 50%
- 企业导师/助教（1-2 人）：权重 30%
- 学生互评（组间互评）：权重 20%

### 评分步骤

1. **路演前**：评委浏览项目 Git 仓库和 README（5 分钟/项目）
2. **路演中**：评委按评分表实时打分，记录亮点和问题
3. **路演后**：评委合议 2 分钟，确认最终分数
4. **统分**：去掉互评的最高分和最低分，取平均值后加权计算

## 五、特殊情形处理

| 情形 | 处理方式 |
|------|----------|
| 团队人数 ≠ 5 人 | 按实际人数评分，但要求每人有独立产出。3-4 人团队允许，2 人团队需完成同等工作量 |
| 路演当天设备故障 | 允许提前录制演示视频（≤ 3 分钟），路演时播放并讲解。扣分：路演表现维度扣 2-3 分 |
| 项目代码无法运行 | 如能证明"昨天还能运行"，允许展示截图/GIF。功能完整性维度扣 3-5 分 |
| 团队成员缺席 | 缺席成员无个人产出则该成员成绩记 0；其余成员正常评分。团队总分不受影响 |
| 互评恶意打分 | 去掉互评最高分和最低分后如仍明显异常，以教师评分为准，取消互评权重 |

## 七、评委守则

1. **一致性**：同一评委对所有项目使用相同评分尺度，避免前松后紧。
2. **证据导向**：打分需有具体证据（Git 提交记录、现场演示、文档内容），不凭印象。
3. **鼓励创新**：功能实现不完美但技术方案有创新（如自研算法、独特工具组合），可在技术实现维度加分 1-3 分。
4. **公平透明**：评分结束后向学生公开评分表（可匿名），允许 24 小时内提出复核申请。
5. **建设性反馈**：每个项目至少给出 1 条具体改进建议，写在评分表备注栏。
