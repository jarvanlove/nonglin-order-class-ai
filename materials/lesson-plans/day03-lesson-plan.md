### 1.3 AI IDE 发展简史（8:40-8:55）

#### 讲解话术（逐字稿）

> （切换到 PPT 第 3-4 页）传统编程是这样的：你打开代码编辑器，一行一行手写代码，遇到不会的就去查文档、搜 Stack Overflow。
>
> AI 编程工具改变了这个流程：你描述需求，AI 直接生成代码；你遇到报错，把报错信息贴给 AI，它告诉你怎么修；你想加新功能，用自然语言描述，AI 帮你改代码。
>
> （切换到第 4 页）我们快速过一下 AI IDE 的发展时间线：
> - 2021 年：GitHub Copilot 发布，首次实现"代码自动补全"
> - 2022 年：Cursor 发布，AI 原生编辑器概念诞生
> - 2023 年：各种 AI 编程工具爆发，国内开始跟进
> - 2024 年：Trae 发布 Builder 模式，"零代码开发"成为可能
> - 2025-2026 年：AI IDE 成为开发者标配

#### 板书提示

```
黑板新增：
AI IDE 演进：
手写代码 → 代码补全(Copilot) → AI编辑器(Cursor) → 零代码开发(Trae Builder)
```

### 1.4 全球 AI IDE 全景对比（8:55-9:10）

> （切换到 PPT 第 5 页）全球 AI IDE 全景。国外三大巨头：

### 1.5 Trae 介绍与 Builder 模式（9:10-9:30）

#### 讲解话术（逐字稿）

> （切换到 PPT 第 6-8 页）Trae 是字节跳动推出的 AI 编程工具。它的核心亮点是 Builder 模式。
>
> （切换到第 7 页）什么是 Builder 模式？
>
> 在这个模式下，你不需要写代码，只需要像跟同事交代任务一样描述需求，Trae 会自动：
> 1. 生成完整的代码文件
> 2. 搭建项目结构
> 3. 帮你运行测试
>
> 对于没有编程基础的同学来说，这是门槛最低的入门方式。
>
> （切换到第 8 页）演示一个 Builder 模式的实际案例：
>
> （在 Trae 中操作）
> 1. 打开 Trae，进入 Builder 模式
> 2. 输入："创建一个简单的 HTML 网页，显示'Hello World'，背景用浅蓝色，文字居中"
> 3. Trae 自动生成代码文件
> 4. 点击"运行"，展示效果
>
> 看到没有？我没有写一行代码，只是用自然语言描述了需求。

### 2.3 API 概念与外部数据接入（10:10-10:30）

#### 讲解话术（逐字稿）

> （切换到 PPT 第 5-6 页）API 是什么？
>
> API（Application Programming Interface）可以理解为"不同软件之间的对话方式"。
>
> 举例：你想让 Trae 做的网页显示实时天气，但 Trae 自己不会"看天"，它需要去问一个专门提供天气数据的网站："北京今天多少度？"这个"问"的方式就是 API。
>
> 你不需要理解 API 的技术细节，只需要知道：Trae 可以通过 API 获取外部数据，让你的项目更有"真实感"。
>
> （切换到第 6 页）在 Trae 中使用 API：
> 1. 找一个免费的天气 API（如和风天气、百度地图）
> 2. 在 Trae 中说："在这个网页上添加一个天气显示模块，调用 [API 地址] 获取当前城市天气"
> 3. Trae 自动生成调用 API 的代码
>
> 注意：课程期间只能使用国内可访问的 API，不要使用需要 VPN 的服务。

#### API 概念类比板书

```
黑板新增：
API = 软件之间的"服务员"

你的网页说："我要天气数据"
API 服务员去厨房（天气服务器）取来给你
你不需要知道厨房怎么做饭

推荐国内免费 API：
- 和风天气（天气数据）
- 百度地图（地理位置）
- 聚合数据（多种数据）
```

### 2.4 Trae 进阶技巧（10:30-10:45）

#### 讲解话术（逐字稿）

> （切换到 PPT 第 4 页）Trae 进阶技巧。
>
> **技巧 1：多文件协作**。大型项目涉及多个文件时，Trae 可以帮你管理项目结构。在 Builder 模式中说："创建一个项目，包含首页、详情页、关于页，使用共享的导航栏和样式文件。"
>
> **技巧 2：迭代修改**。不需要一次性描述完美需求。先让 Trae 做一个基础版本，然后逐步添加功能："在现有页面上加一个搜索框"、"把表格改成卡片式布局"。
>
> **技巧 3：代码解释**。看到 Trae 生成的代码不懂？选中代码，问 Trae："这段代码是做什么的？"它会用自然语言解释。
>
> **技巧 4：错误修复**。遇到报错，把报错信息复制给 Trae，说："出现了这个报错，请帮我修复。"Trae 会分析原因并给出修复方案。

#### 进阶技巧板书


### 3.3 QClaw 模块开发（13:05-13:30）

#### 操作指令

**步骤 1：确定采集目标**
1. 根据选题确定需要搜集的信息类型
2. 列出关键词列表（至少 5 个）
3. 确定采集数量（建议 10-20 条）

**步骤 2：编写采集 Prompt**
```
【角色】你是一位 [领域] 信息搜集专家
【任务】请搜集 [时间范围] 内关于 [主题] 的信息
【要求】
1. 至少 [数量] 条记录
2. 每条记录包含：标题、来源、日期、摘要、链接、分类
3. 输出格式：JSON
4. 分类维度：[列出分类]
5. 提供来源链接
```

**步骤 3：执行采集并保存**
1. 在 QClaw 中输入采集 Prompt
2. 等待执行完成
3. 检查结果，不满足要求则补充指令
4. 保存输出到 `data/raw/` 目录

#### 讲师/助教动作

- 讲师在讲台观察整体进度
- 助教全场巡视，解答操作问题
- 对已完成的学生，分配进阶任务："尝试用不同的关键词再采集一批数据"

#### 常见错误预判

| 错误 | 现象 | 解决步骤 |
|------|------|---------|
| QClaw 采集结果为空 | 返回"未找到相关信息" | 1. 检查采集源是否可访问 2. 简化指令 3. 换关键词 |
| 输出格式不符合约定 | JSON 字段名不对 | 在 Prompt 中明确指定字段名和格式 |
| 采集数量不够 | 只返回 2-3 条 | 增加关键词数量，放宽筛选条件 |

### 3.4 WorkBuddy 模块开发（13:30-13:55）

#### 操作指令

**步骤 1：读取 QClaw 输出**
1. 打开 WorkBuddy，进入 Craft 模式
2. 上传 QClaw 输出的 JSON 文件
3. 输入指令："请读取这个 JSON 文件，告诉我有多少条记录，字段有哪些"

**步骤 2：数据清洗**
```
请帮我清洗这份数据：
1. 删除完全重复的记录（根据 title 和 source 判断）
2. 填充空值：title 为空则标记为"无标题"，summary 为空则标记为"暂无摘要"
3. 统一日期格式为 YYYY-MM-DD
4. 按 category 分类统计
5. 将清洗后的数据保存到 data/processed/summary.json
```

**步骤 3：生成报告（可选）**
```
请基于清洗后的数据，生成一份数据分析报告，包含：
1. 数据概览（总条数、分类统计）
2. 时间分布分析
3. 来源分布分析
4. 保存为 output/analysis_report.docx
```

#### 讲师/助教动作

- 助教全场巡视，解答操作问题
- 重点检查数据清洗规则是否合理
- 确认输出文件格式是否符合接口约定

#### 常见错误预判

| 错误 | 现象 | 解决步骤 |
|------|------|---------|
| JSON 文件导入 WorkBuddy 后字段错位 | title 列显示的是 content 内容 | 1. 检查字段名 2. 调整映射 3. 改用 CSV |
| WorkBuddy 生成的 Word 文档打不开 | 双击 docx 报错"文件已损坏" | 1. 关 Word 重试 2. 简化格式 3. 测试短内容 |
| 清洗后数据丢失 | 记录数减少过多 | 检查去重条件是否太严格 |

### 3.5 模块联调测试（13:55-14:15）

#### 联调测试步骤

**第一步：确认数据出口（QClaw 侧）**
1. 在 QClaw 中完成一次完整采集
2. 检查输出文件：存在、不是乱码、是合法 JSON
3. 确认字段名和接口约定一致

**第二步：确认数据入口（WorkBuddy 侧）**
1. 打开 WorkBuddy，上传 QClaw 输出文件
2. 导入后预览数据，确认字段名和记录数
3. 执行清洗，检查输出文件

**第三步：端到端测试**
1. 清空 data/ 目录，重新开始
2. QClaw 执行完整采集
3. WorkBuddy 读取新文件并执行清洗
4. 检查 `data/processed/summary.json`：内容完整、格式正确、数据准确

#### 联调验收标准

| 验收项 | 通过标准 | 检查方式 |
|--------|---------|---------|
| QClaw 自动采集 | 运行一次采集指令，成功返回结构化数据 | 看 QClaw 对话窗口 |
| 保存为文件 | 桌面出现 `qclaw-output.json` | 看桌面文件 |
| 数据格式符合约定 | 字段包含约定的 key | 用记事本打开检查 |
| 至少 5 条样本 | 文件内记录数 >= 5 | 数行数 |
| WorkBuddy 读取 | 能加载数据 | 看 WorkBuddy 日志 |
| 数据清洗完成 | 字段整齐、无乱码 | 打开输出文件目视检查 |
| 生成目标文档 | 桌面出现报告文件 | 看桌面文件 |
