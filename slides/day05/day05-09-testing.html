测试与文档：让项目从"能用"到"好用"

三级测试体系

功能测试：核心流程是否跑通

列出项目所有功能点，每个至少测试一次正常流程。例如：点击搜索按钮 → 输入关键词 → 确认结果正确显示。

边界测试：异常情况处理

测试极端情况：空输入、超长输入、特殊字符、网络断开、数据文件缺失。好的项目应该在异常时给出友好提示，而不是崩溃。

用户验收测试（UAT）

找另一组同学当"用户"，观察他们是否能独立完成任务。记录困惑点和操作卡点——这些就是需要优化的地方。

Bug 修复优先级

P0

阻断性 Bug：页面空白、按钮无反应、数据读取失败——必须立即修复

P1

功能性 Bug：筛选有时无效、数据显示不完整——尽量修复

P2

体验性 Bug：按钮颜色不好看、加载慢——有余力再修

项目文档：README 撰写模板

README 是项目的"门面"，让其他人（包括明天的你自己）能快速理解这个项目是做什么的、怎么运行。

# 项目名称

## 安装与运行

// 步骤 1、2、3...

## 团队成员

// 分工表

文档撰写技巧

1. 先写项目简介：用一句话说明项目解决什么问题。

2. 再列功能清单：不要写太多，3-5 个核心功能即可。

3. 技术方案要画数据流：用文字或箭头说明 QClaw → WorkBuddy → Trae 的流转过程。

4. 安装步骤要具体：不要写"打开 Trae"，要写"双击桌面 Trae 图标，等待加载完成"。
