## 三、分组与角色分配（14:30 - 14:50，20 分钟）

### 3.1 分组策略

**【讲师话术】**> "30 人分成 5-6 组，每组 5-6 人。分组原则：混合搭配。"

**分组原则**：
1. **计算机与非计算机混合**：每组至少 2 名计算机专业学生（技术骨干），至少 2 名非计算机学生（产品/调研视角）
2. **性别平衡**：尽量男女搭配
3. **自愿 + 微调**：先让学生自由组合，讲师根据原则微调
4. **指定组长**：每组选 1 名组长，负责进度把控和对外沟通

**【讲师话术】**> "非计算机的同学不要觉得自己没用。你们的专业背景——农林、生物、经管——恰恰是项目的灵魂。计算机同学能写代码，但你们更懂业务需求。"

### 3.2 角色分配建议

每组 5-6 人，建议以下角色分配：

| 角色 | 人数 | 职责 | 对应工具 | 适合谁 |
|------|------|------|---------|--------|
| **组长/项目经理** | 1 人 | 进度管理、需求把控、对外汇报 | 全部 | 有组织能力的任何人 |
| **QClaw 模块负责人** | 1 人 | 信息搜集、数据获取、自动化流程 | QClaw | 善于搜索、有耐心的人 |
| **WorkBuddy 模块负责人** | 1 人 | 数据处理、文档生成、报告输出 | WorkBuddy | 细心、Excel 熟练者 |
| **Trae 模块负责人** | 1-2 人 | 核心功能开发、界面实现、联调 | Trae | 计算机专业或愿意学代码的人 |
| **产品/测试** | 1 人 | 需求文档、用户测试、Bug 记录 | 无特定 | 善于沟通、有产品思维的人 |

**【讲师话术】**> "角色不是固定的。如果 QClaw 负责人提前完成了任务，可以去帮 Trae 的同学。我们追求的是项目成功，不是个人 KPI。"

## 四、需求拆解方法（14:50 - 15:30，40 分钟）

### 4.1 为什么要拆需求？

**【讲师话术】**> "拿到项目题目后，第一反应往往是'这个好复杂，怎么做啊？'需求拆解就是把这个复杂问题，切成一块块能下嘴的小蛋糕。"

### 4.2 拆解三步法

#### 第一步：用户故事（Who + What + Why）

**定义**：从用户视角描述需求，格式为：

```
作为一个 [角色]，
我想要 [功能]，
以便 [价值/目的]。
```

**示例**（以"农林信息聚合助手"为例）：

```
1. 作为一个农林专业学生，我想要自动搜集最新的农业政策，以便及时了解行业动态。
2. 作为一个课题组助理，我想要把零散的信息整理成结构化周报，以便向导师汇报。
3. 作为一个求职者，我想要一个网页查看整理好的信息，以便快速浏览和搜索。
```

**【讲师话术】**> "写用户故事的时候，不要想'我们要做什么功能'，要想'用户需要什么'。这个视角转换非常重要。"

#### 第二步：功能列表（把故事翻译成功能）

将用户故事拆解为具体功能点：

| 用户故事 | 功能点 | 负责工具 | 优先级 |
|---------|--------|---------|--------|
| 搜集农业政策 | 1.1 配置 QClaw 定时搜索关键词<br>1.2 自动保存搜索结果到本地 | QClaw | P0（必须） |
| 整理成周报 | 2.1 读取 QClaw 保存的原始数据<br>2.2 按类别分类整理<br>2.3 生成格式化的 Word/Excel 报告 | WorkBuddy | P0 |
| 网页查看信息 | 3.1 展示信息列表（支持分类筛选）<br>3.2 搜索功能<br>3.3 响应式布局（手机也能看） | Trae | P0 |
| 额外功能 | 4.1 信息收藏功能<br>4.2 导出 PDF | Trae | P1（加分） |

**优先级说明**：
- **P0**：必须完成，不做项目就不成立
- **P1**：加分项，P0 做完后有时间再做
- **P2**：理想功能，大概率来不及做

**【讲师话术】**> "记住一个原则：先完成，再完美。Day 6 路演时，一个功能完整但简陋的 Demo，远好于一个功能残缺但花哨的 Demo。"

#### 第三步：技术方案（How）

为每个功能点设计技术实现方案：

**示例**：

```markdown
## 技术方案：农林信息聚合助手

### 模块一：QClaw 信息搜集
- **输入**：配置好的关键词列表（如"智慧农业""农产品价格""农业政策 2026"）
- **处理**：QClaw 定时执行搜索任务，抓取网页摘要
- **输出**：JSON/txt 文件，保存到项目目录 `/data/raw/` 下
- **负责人**：QClaw 模块负责人

### 模块二：WorkBuddy 数据处理
- **输入**：`/data/raw/` 下的原始数据文件
- **处理**：
  1. 读取原始数据
  2. 按主题分类（政策/市场/技术）
  3. 生成 Excel 汇总表（标题、来源、日期、摘要、链接）
  4. 生成 Word 周报文档（带格式）
- **输出**：`/data/processed/weekly_report.docx` 和 `summary.xlsx`
- **负责人**：WorkBuddy 模块负责人

### 模块三：Trae 网页开发
- **输入**：`/data/processed/summary.xlsx` 中的结构化数据
- **处理**：
  1. 用 Builder 模式生成网页骨架（HTML + CSS + JS）
  2. 实现信息列表展示
  3. 实现分类筛选功能
  4. 实现搜索功能（前端过滤）
- **输出**：可运行的网页，部署到本地或 GitHub Pages
- **负责人**：Trae 模块负责人

### 数据流转图
```
[QClaw 搜索] → raw_data.json → [WorkBuddy 处理] → summary.xlsx → [Trae 网页] → 用户查看
```
```

**【讲师话术】**> "技术方案不需要写得很复杂。关键是把'谁做什么、输入是什么、输出是什么'说清楚。这就像做菜前先列菜单和备料清单。"

### 4.3 需求拆解实操

**分组任务**（20 分钟）：

1. 每组拿到项目任务书后，用上述三步法拆解需求
2. 组长在白板/纸上画出：用户故事 → 功能列表 → 技术方案
3. 讲师和助教巡视，逐组辅导

**检查清单**：
- [ ] 是否写出了至少 3 个用户故事？
- [ ] 功能列表是否覆盖了三个工具？
- [ ] 每个功能是否标了优先级（P0/P1/P2）？
- [ ] 技术方案是否明确了输入/处理/输出？
- [ ] 数据流转是否通畅（QClaw → WorkBuddy → Trae）？

├── docs/                   # 文档目录
│   ├── requirement.md      # 需求文档
│   ├── tech-design.md      # 技术方案
│   └──分工表.md             # 分工与进度

│   └── js/

### 6.4 跨模块数据接口约定

**关键**：三个模块之间需要约定数据格式，确保能顺畅对接。

**推荐的数据接口**：

**QClaw → WorkBuddy**：
- 格式：JSON 或 CSV
- 必需字段：`title`（标题）、`source`（来源）、`date`（日期）、`summary`（摘要）、`url`（链接）、`category`（类别）
- 文件位置：`data/raw/news_YYYYMMDD.json`

**WorkBuddy → Trae**：
- 格式：JSON 或 Excel 转 JSON
- 必需字段：同上，增加 `processed_date`（处理日期）
- 文件位置：`data/processed/summary.json`

**示例 JSON 结构**：
```json
[
  {
    "title": "2026年中央一号文件发布",
    "source": "农业农村部官网",
    "date": "2026-02-01",
    "summary": "文件强调智慧农业发展和粮食安全保障...",
    "url": "https://www.moa.gov.cn/...",
    "category": "政策"
  },
  {
    "title": "小麦价格连续三周上涨",
    "source": "农产品网",
    "date": "2026-04-25",
    "summary": "受气候影响，主产区小麦价格环比上涨 5%...",
    "url": "https://www.agri.cn/...",
    "category": "市场"
  }
]
```

**【讲师话术】**> "这个 JSON 格式是三个模块之间的'合同'。QClaw 输出的数据要符合这个格式，WorkBuddy 处理时也要保持这个格式，Trae 读取时按这个格式解析。格式统一了，模块之间才能'对话'。"

## 七、常见错误排查

### 7.1 Trae 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| Builder 模式没有生成文件 | 模型响应超时 | 换 DeepSeek 模型重试，或简化提示词 |
| 生成的代码运行报错 | AI 生成的代码有语法错误 | 把报错信息复制到 Chat 面板，让 AI 修复 |
| 网页样式错乱 | CSS 冲突或路径错误 | 检查 CSS 文件路径，或用内联 CSS |
| Trae 卡顿/无响应 | 电脑内存不足 | 关闭其他程序，或重启 Trae |
| 模型返回英文 | 提示词是英文或模型默认英文 | 在提示词中明确说"请用中文" |

### 7.2 QClaw 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 搜索无结果 | 关键词太冷门 | 换更通用的关键词 |
| 保存文件失败 | 路径权限问题 | 保存到桌面或文档目录，再手动移动到项目文件夹 |
| 数据格式混乱 | AI 输出不稳定 | 在提示词中明确要求"输出 JSON 格式" |

### 7.3 WorkBuddy 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 读取 Excel 失败 | 文件格式不兼容 | 另存为 .xlsx 格式再试 |
| 生成的报告格式错乱 | 模板定义不清 | 先做一个简单的模板，逐步增加复杂度 |
| 中文字符显示乱码 | 编码问题 | 确保文件保存为 UTF-8 编码 |

### 7.4 跨模块常见问题

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| Trae 读不到 WorkBuddy 的数据 | 文件路径错误或格式不对 | 检查文件路径，确认 JSON 格式正确 |
| QClaw 的数据 WorkBuddy 处理不了 | 字段名不一致 | 对照数据接口约定，统一字段名 |
| 联调时某个模块没输出 | 该模块还没完成 | 先用假数据（mock data）代替，确保其他模块能继续开发 |

**Mock 数据示例**：
```json
[
  {
    "title": "测试数据-农业政策",
    "source": "测试来源",
    "date": "2026-04-27",
    "summary": "这是测试数据，用于联调",
    "url": "https://example.com",
    "category": "政策"
  }
]
```

**【讲师话术】**> "联调时如果某个模块还没完成，不要干等。用假数据先跑起来，这叫' mock 数据'。等真实数据来了，替换一下就行。这是专业开发的标准做法。"
