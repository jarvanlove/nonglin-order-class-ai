### 2.2 数据流联调 Checklist（逐项验收）

**讲师话术**：
> "联调的核心是'数据流'——数据从 QClaw 流出，经过 WorkBuddy 处理，最终流入 Trae 展示。我们按这个顺序逐项检查。"

#### 第一关：QClaw → 数据输出（验收人：负责 QClaw 的同学）

- [ ] QClaw 已完成信息搜集任务（如：搜集了 10 条农产品价格信息）

- [ ] 输出文件保存在全组可访问的位置（如项目文件夹的 `data/` 目录）
- [ ] 文件编码为 UTF-8，无乱码
- [ ] 数据结构包含 Trae 模块需要的所有字段（如：名称、价格、产地）

**示例 JSON 输出格式**：
```json
[
  {
    "name": "东北大米",
    "category": "粮食",
    "origin": "黑龙江",
    "price": "5.5元/斤",
    "updateTime": "2026-04-27"
  }
```

#### 第二关：WorkBuddy → 数据处理（验收人：负责 WorkBuddy 的同学）

- [ ] WorkBuddy 能正确读取 QClaw 的输出文件

- [ ] 数据转换完成（如：Excel → JSON、多表合并）
- [ ] 处理后的数据保存到新文件，不覆盖原始数据
- [ ] 输出文件路径已告知 Trae 负责同学

**WorkBuddy 处理要点**：
- 使用 WorkBuddy 的"批量处理"功能自动化清洗
- 复杂转换可导出为脚本，方便重复执行
- 处理日志要保存，便于排查问题

#### 第三关：Trae → 数据展示（验收人：负责 Trae 的同学）

- [ ] Trae 模块能读取 WorkBuddy 处理后的数据文件
- [ ] 数据正确显示在界面上（无 undefined、无空白）
- [ ] 界面交互功能正常（筛选、搜索、点击等）
- [ ] 数据更新后，刷新页面能显示最新内容
- [ ] 错误处理：数据文件缺失时显示友好提示

**Trae 读取本地 JSON 的代码模板**：
```javascript
fetch('./data/products.json')
  .then(response => {
    if (!response.ok) {
      throw new Error('数据文件加载失败');
    }
    return response.json();
  })
  .then(data => {
    console.log('加载成功，共', data.length, '条数据');
    renderProducts(data);
  })
  .catch(error => {
    console.error(error);
    document.getElementById('error-msg').textContent = '数据加载失败，请检查数据文件是否存在';
  });
```

### 2.3 联调问题快速响应机制

**讲师话术**：
> "联调时遇到问题不要闷头搞，按这个流程来：先定位是哪一块的问题，再找对应负责人，解决不了再喊讲师。"

**问题定位三步法**：
1. **看数据文件**：QClaw 输出的文件内容对吗？
2. **看中间结果**：WorkBuddy 处理后的文件内容对吗？
3. **看前端表现**：Trae 读取数据后渲染正确吗？

**常见联调问题速查**：

| 现象 | 定位 | 解决方向 |
|------|------|---------|
| Trae 页面显示空白 | 数据未正确加载 | 检查文件路径、JSON 格式、CORS 限制 |
| 显示 undefined | 字段名不匹配 | 对比 QClaw 输出字段名和 Trae 读取字段名 |
| 数据条数不对 | WorkBuddy 过滤/去重过度 | 检查 WorkBuddy 处理规则 |
| 中文乱码 | 编码问题 | 确保所有环节使用 UTF-8 |
| 刷新后数据不更新 | 浏览器缓存 | 按 Ctrl+F5 强制刷新 |

**讲师话术**：
> "联调完了不等于项目完了，还要测试。测试分三级，从里到外逐步验证。"

#### 第一级：功能测试（单元级）

**目标**：每个独立功能点都能正常工作

**测试方法**：

#### 第二级：集成测试（模块间）

- 找另一组同学当"用户"，不看代码，只按界面提示操作

### 4.2 Bug 修复优先级

**讲师话术**：
> "测试一定会发现 Bug，但 Bug 有轻重缓急。不要在一个小样式问题上纠结半小时，先修阻断性的。"

#### P0：阻断性 Bug（必须立即修复）

**定义**：导致核心功能完全不可用，或导致程序崩溃

**示例**：
- 页面打开空白，无法显示任何内容
- 点击核心按钮无任何反应
- 数据文件读取失败导致整个流程中断
- 程序报错无法继续运行

**修复原则**：
- 全组优先处理，必要时暂停其他工作
- 如果 15 分钟内无法修复，考虑降级方案（如用静态数据代替动态数据）

#### P1：功能性 Bug（尽量修复）

**定义**：功能可用，但结果不正确或部分场景失效

**示例**：
- 筛选功能有时有效有时无效
- 数据显示不完整（缺字段、缺条目）
- 搜索功能对中文支持不好
- 数据排序不正确

**修复原则**：
- 在 P0 修复完成后处理
- 评估修复成本，如果太复杂且不影响演示，可记录为"已知问题"

#### P2：体验性 Bug（有余力再修）

**定义**：不影响功能，但影响用户体验

**示例**：
- 按钮颜色不好看
- 页面加载慢（但还能接受）
- 提示文字不够友好
- 移动端显示有点错位

**修复原则**：
- 最后处理，如果时间不够可以不改
- 文档中注明"已知优化点"即可

## 六、常见错误排查（16:15 并行进行，贯穿下午）

### 错误 1：QClaw 输出文件 Trae 读不到

**现象**：Trae 中 `fetch('./data/xxx.json')` 报错 404
**排查步骤**：
1. 确认文件确实存在于 `data/` 目录
2. 确认文件名大小写一致（Windows 不区分大小写，但部分服务器区分）
3. 确认 Trae 的 Live Server 根目录是项目根目录，不是某个子文件夹
4. 尝试用完整相对路径：`./data/xxx.json` 或 `data/xxx.json`

### 错误 2：JSON 数据解析失败

**现象**：`JSON.parse` 报错 或 `response.json()` 失败
**排查步骤**：
1. 用文本编辑器打开 JSON 文件，检查是否格式正确（末尾无逗号、引号成对）
2. 确认文件编码为 UTF-8，无 BOM 头
3. 在浏览器 F12 → Network 中查看实际返回的内容
4. 如果 JSON 文件有注释，删除注释（标准 JSON 不支持注释）

### 错误 3：WorkBuddy 处理后的数据格式不对

**现象**：Trae 读取后字段名变了，或数据结构嵌套层级不对
**排查步骤**：
1. 对比 QClaw 原始输出、WorkBuddy 处理后输出、Trae 期望输入三个文件
2. 检查 WorkBuddy 的导出设置，确认字段映射正确
3. 在 Trae 的 JS 中先 `console.log(data)` 查看实际结构
4. 调整 Trae 代码中的字段访问方式，或让 WorkBuddy 重新导出正确格式

### 错误 4：界面显示数据但排版错乱

**现象**：数据能显示，但卡片重叠、文字溢出、按钮错位
**排查步骤**：
1. 检查 CSS 中是否有固定宽度/高度导致内容溢出
2. 检查父容器是否有 `display: flex` 或 `grid` 布局设置
3. 在 F12 → Elements 中选中错乱元素，查看实际应用的 CSS 规则
4. 让 Builder 模式修复："卡片内容过长时自动换行，不要溢出"

### 错误 5：跨模块协作时文件被覆盖

**现象**：WorkBuddy 输出文件后，QClaw 重新运行又覆盖了它
**排查步骤**：
1. 约定文件命名规范：QClaw 输出 `raw_data.json`，WorkBuddy 输出 `processed_data.json`
2. 各模块读写不同文件，不要操作同一个文件名
3. 在 README 中明确记录每个文件的职责和生成方式

### 错误 6：联调时某同学电脑能跑，另一台不能

**现象**：同一套代码，在不同电脑上表现不同
**排查步骤**：
1. 检查浏览器版本，建议使用 Chrome 或 Edge 最新版
2. 检查 Trae 版本是否一致
3. 检查本地数据文件是否已同步（可用 U 盘或微信传文件）
4. 检查是否有浏览器插件拦截了网络请求

### 错误 7：演示时网络 API 请求失败

**现象**：调用和风天气/聚合数据 API 时失败
**排查步骤**：
1. 检查网络连接是否正常
2. 检查 API Key 是否已过期或达到调用上限
3. 准备降级方案：使用本地静态数据文件代替实时 API
4. 演示前提前测试 API 可用性

### 错误 8：Git 合并冲突（如果使用了 Git）

**现象**：`git pull` 或 `git merge` 提示冲突
**排查步骤**：
1. 打开冲突文件，搜索 `<<<<<<< HEAD` 标记
2. 手动选择保留哪部分代码，或合并两部分
3. 删除冲突标记线
4. 重新提交：`git add . && git commit -m "解决合并冲突"`
5. **预防**：各组员尽量修改不同文件，不要同时改同一个文件
