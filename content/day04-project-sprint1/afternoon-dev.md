## 一、核心功能开发 Checklist（14:00-15:30）

### 1.2 办公自动化模块（WorkBuddy 负责）

| # | 验收项 | 通过标准 | 检查方式 |
|---|--------|---------|---------|

| 1.2.2 | 数据清洗与格式化完成 | 输出文件字段整齐、无乱码、日期格式统一 | 打开输出文件目视检查 |
| 1.2.3 | 生成目标文档（Word/Excel/PDF 至少一种） | 桌面出现生成的报告文件 | 看桌面文件 |
| 1.2.4 | 文档内容包含项目核心信息 | 报告中有数据、有结论、有来源，不是空模板 | 打开文档检查 |
| 1.2.5 | （进阶）使用模板生成文档 | 文档格式与预设模板一致（字体、颜色、表头） | 对比模板和输出文件 |
| 1.2.6 | （进阶）多 Agent 协作运行 | 两个以上 Agent 按顺序执行，无冲突 | 看 WorkBuddy 执行日志 |

### 1.3 模块间接口约定（两组必须共同遵守）

**数据格式约定**：
```json
{
  "version": "1.0",
  "generated_by": "QClaw",
  "generated_at": "2026-04-27T14:30:00+08:00",
  "data": [
    {
      "id": 1,
      "title": "信息标题",
      "content": "信息正文或摘要",
      "source": "来源网址或文件名",
      "category": "分类标签",
      "date": "2026-04-27",
      "confidence": 0.95
    }
  ]
}
```

**文件位置约定**：
- QClaw 输出：`C:\Users\[用户名]\Desktop\project-data\qclaw-output.json`
- WorkBuddy 读取：配置为读取上述路径
- WorkBuddy 输出：`C:\Users\[用户名]\Desktop\project-data\final-report.docx`

**字段命名约定**：
- 全部使用英文小写 + 下划线，禁止中文字段名
- 日期统一格式：`YYYY-MM-DD`
- 空值用 `""`（空字符串），不要用 `null` 或删除字段

## 二、QClaw → WorkBuddy 数据联调步骤（15:45-17:30）

联调是 Day 4 下午的核心任务，必须按以下步骤逐项执行。

### 2.1 第一步：确认数据出口（QClaw 侧，15 分钟）

**操作步骤**：
1. 在 QClaw 中完成一次完整的信息采集
2. 在指令中明确要求保存到文件："请将以上结果保存到桌面的 project-data 文件夹，文件名为 qclaw-output.json，使用 UTF-8 编码"
3. 如果 QClaw 无法直接保存文件，手动复制对话中的 JSON 内容
4. 打开桌面 `project-data/qclaw-output.json`，确认：
   - 文件存在
   - 内容不是乱码
   - 是合法的 JSON（可用在线 JSON 校验工具，或让 Trae 帮忙检查）

**通过标准**：`qclaw-output.json` 存在且能被 JSON 解析器正常读取。

**常见错误**：
- QClaw 输出的是 Markdown 代码块包裹的 JSON，不是纯 JSON → 手动删除 ```json 和 ``` 标记
- 中文字符被转成了 Unicode（如 中文）→ 这是合法的，WorkBuddy 能正常读取
- 文件保存到了奇怪的路径 → 在 QClaw 指令中明确指定"保存到桌面"

### 2.2 第二步：确认数据入口（WorkBuddy 侧，15 分钟）

**操作步骤**：
1. 打开 WorkBuddy，进入"数据导入"或"文件处理"模块
2. 点击"选择文件"，导航到 `Desktop/project-data/qclaw-output.json`
3. 如果 WorkBuddy 支持 JSON 直接导入，选择 JSON 格式
4. 如果不支持 JSON，先用 Trae 写一个小脚本把 JSON 转成 CSV，或让 QClaw 直接输出 CSV 格式
5. 导入后，在 WorkBuddy 中预览数据，确认字段名和记录数正确

**通过标准**：WorkBuddy 能显示 qclaw-output.json 中的数据，字段不错位。

**常见错误**：
- WorkBuddy 不支持 JSON → 解决方案：让 QClaw 输出 CSV，或用 Trae 写格式转换脚本
- 字段映射错误（如 title 被映射成了 content）→ 在 WorkBuddy 中手动调整字段映射关系
- 编码错误，中文显示乱码 → 确认文件是 UTF-8，必要时用记事本"另存为"重新选 UTF-8

### 2.3 第三步：设计数据转换规则（20 分钟）

**操作步骤**：
1. 两组同学一起对照项目需求，明确"QClaw 输出什么"和"WorkBuddy 需要什么"
2. 在纸上画出数据转换流程：
   ```
   qclaw-output.json
      ↓
   [数据清洗] 去重、去空、格式统一
      ↓
   [数据转换] 按模板要求重组字段
      ↓
   [文档生成] 填入 Word/Excel 模板
      ↓
   final-report.docx
   ```
3. 在 WorkBuddy 中配置转换规则：
   - 打开"数据处理" > "转换规则"
   - 添加规则：如果 `confidence < 0.8`，标记为"待核实"
   - 添加规则：如果 `category == "新闻"`，归入"动态汇总"表格
   - 添加规则：`date` 字段统一格式化为 `YYYY年MM月DD日`

**通过标准**：转换规则配置完成，用测试数据跑一遍，输出符合预期。

### 2.4 第四步：端到端联调测试（30 分钟）

**操作步骤**：
1. 清空 `project-data` 文件夹，重新开始
2. QClaw 执行完整采集流程，生成新的 `qclaw-output.json`
3. WorkBuddy 自动（或手动触发）读取新文件
4. WorkBuddy 执行数据清洗、转换、文档生成
5. 检查 `final-report.docx`：
   - 内容是否完整
   - 格式是否正确
   - 数据是否准确（抽查 2-3 条与 QClaw 原始输出对比）

**通过标准**：从 QClaw 采集到 WorkBuddy 生成报告，全程无需手动复制粘贴，数据准确无误。

| 字段缺失 | 删除某条记录的 `category` 字段 | WorkBuddy 填入"未分类"或留空，不崩溃 |
| 文件不存在 | 改名或删除输入文件后运行 | WorkBuddy 提示"找不到输入文件" |

| 大数据量 | 放入 50 条以上记录测试 | WorkBuddy 处理时间合理，不卡死 |

**操作步骤**：
1. 用 mock 数据逐一测试上述边界情况
2. 每测完一个，在组内共享文档中记录结果
3. 发现未处理的边界情况，立即修复配置或脚本

## 四、MVP 标准定义（最低可接受产出）

Day 4 结束时的最低标准，达不到的小组需要晚上加班或调整 Day 5 计划。

### 4.1 必须达成（缺一不可）

1. **QClaw 能采集信息**：至少成功采集 5 条结构化数据，保存为文件
2. **WorkBuddy 能处理数据**：读取 QClaw 输出，生成一份可读的文档（Word/Excel/PDF）
3. **数据能流动**：从 QClaw 采集到 WorkBuddy 生成文档，全程不需要手动复制粘贴中间结果
4. **文档有内容**：生成的文档不是空模板，包含实际采集的数据和至少一句结论/摘要

### 4.2 建议达成（加分项，不影响及格）

1. 使用了 QClaw Skill 完成采集
2. WorkBuddy 使用了模板生成文档
3. 配置了多 Agent 协作
4. 处理了异常数据（空值、低置信度等）
5. 测试了 2 个以上边界情况

### 4.3 未达成 MVP 的应急方案

如果 17:00 仍有组未达成 MVP：
1. 讲师/助教直接介入，帮该组完成最小数据流
2. 允许该组使用 mock 数据代替真实采集，确保 WorkBuddy 模块能演示
3. 该组 Day 5 优先完成 Trae 模块，QClaw/WorkBuddy 部分用简化版
4. 记录到讲师备注，路演时酌情考虑难度系数

## 五、常见错误排查手册（至少 10 个问题）

以下问题按发生频率排序，助教随身携带此页，学生问问题时直接对号入座。

### 问题 1：QClaw 采集结果为空 / 返回"未找到相关信息"

**排查步骤**：
1. 检查采集源是否可访问：用浏览器直接打开目标网址，确认能正常显示
2. 简化指令：不要一次提太多要求，先让 QClaw"访问这个网页，告诉我标题是什么"
3. 换采集方式：网页不可用的话，改用"上传文件分析"或"搜索关键词"
4. 检查 Skill 权限：如果用了 Skill，确认 Skill 有网络访问权限

### 问题 2：JSON 文件导入 WorkBuddy 后字段错位

**排查步骤**：
1. 用记事本打开 JSON，确认字段名和预期一致（区分大小写）
2. 检查 WorkBuddy 的字段映射设置：手动指定 `title` 对应"标题"列
3. 如果 JSON 嵌套层级太深，先用 Trae 写脚本展平（flatten）
4. 改用 CSV 格式：让 QClaw 输出 CSV，WorkBuddy 对 CSV 的支持通常更好

### 问题 3：WorkBuddy 生成的 Word 文档打不开 / 提示损坏

**排查步骤**：
1. 检查文件是否被占用：关闭所有打开的 Word 窗口，重新生成
2. 模板问题：如果用了模板，检查模板文件本身是否能正常打开
3. 内容过长：某条记录内容太长可能导致文档生成失败，先测试短内容
4. 路径问题：输出路径不要包含特殊字符，尽量用英文路径

### 问题 4：QClaw 微信消息发不出去 / 收不到回复

**排查步骤**：
1. 检查微信在线状态：手机和电脑微信都要在线
2. 检查 QClaw 绑定状态：重新扫码绑定
3. 消息频率限制：短时间内发太多消息可能触发微信风控，间隔 5 秒以上
4. 群聊权限：如果是群消息，确认 QClaw 在群里，且没有被设为"消息免打扰"

### 问题 5：WorkBuddy Agent 不触发 / 没有自动执行

**排查步骤**：
1. 检查触发条件：文件监控类触发，确认监控路径正确且有新文件放入
2. 检查 Agent 开关：确认 Agent 状态是"启用"不是"暂停"
3. 检查执行日志：WorkBuddy 有日志面板，看是否有错误信息
4. 手动触发测试：先用手动执行确认任务本身没问题，再排查自动触发

### 问题 6：Trae 写的格式转换脚本运行报错

**排查步骤**：
1. 确认语言环境：Node 脚本需要 `node` 命令，Python 脚本需要 `python` 命令
2. 把报错信息复制到 Trae Chat 模式，让 AI 修复
3. 检查文件路径：脚本中的路径是否用了正确的斜杠（Windows 用 `\\` 或 `/`）
4. 缺少依赖：Python 的 `json`/`csv` 是内置模块，但如果用了 `pandas` 需要安装

### 问题 7：数据中有乱码 / 中文显示为问号

**排查步骤**：
1. 统一编码：所有文件保存为 UTF-8，QClaw 输出时指定"UTF-8 编码"
2. WorkBuddy 导入时选择正确的编码格式
3. 如果已经乱码，用记事本"另存为"重新选择 UTF-8 覆盖保存
4. Excel 打开 CSV 时默认用 GBK，需要选择"数据 > 从文本/CSV 导入"手动选 UTF-8

### 问题 8：QClaw 输出格式每次都不一样

**排查步骤**：
1. 在指令中明确指定输出格式："请严格按照以下 JSON 格式输出..."
2. 给出示例：在指令中贴一个期望的输出样例
3. 使用 Skill：Skill 的输出通常比自由对话更稳定
4. 后处理：如果 QClaw 输出不稳定，用 Trae 写脚本规范化

### 问题 9：WorkBuddy 处理速度慢 / 卡死

**排查步骤**：
1. 减少数据量：先测试 5-10 条，确认流程正确后再处理全部
2. 关闭不必要的 Agent：只保留当前需要的 Agent，其他的暂停
3. 重启 WorkBuddy：长时间运行后可能有内存泄漏，重启即可
4. 简化文档模板：模板中不要放太多复杂格式（图片、复杂表格）

### 问题 10：组内代码/配置文件互相覆盖

**排查步骤**：
1. 立刻使用版本控制：在 Trae 中初始化 Git（左侧源代码管理 > 初始化仓库）
2. 分工明确：谁改哪个文件提前约定，不要多人同时改同一个配置
3. 定时同步：每完成一个功能点，组内互相展示成果，确保大家都在同一版本
4. 备份策略：重要文件改之前复制一份 `.bak`，改坏了能回退

### 问题 11：QClaw Skill 返回的结果和预期不符

**排查步骤**：
1. 查看 Skill 说明：每个 Skill 都有使用文档，确认参数传对了
2. 简化测试：先用最简单的参数测试，逐步增加复杂度
3. 反馈问题：如果是 Skill 本身的问题，在 ClawHub 反馈或换别的 Skill
4. 兜底方案：不用 Skill，用基础指令完成同样任务

### 问题 12：WorkBuddy 模板中的 `{{字段}}` 没有被替换

**排查步骤**：
1. 检查字段名拼写：模板中的 `{{姓名}}` 和数据源中的 `name` 是否映射正确
2. 检查括号格式：必须是双大括号 `{{}}`，不是单括号 `{}` 或中文括号
3. 检查数据源：确认该字段在数据中存在且不为空
4. 手动测试：在 WorkBuddy 中先做一个最简单的模板（只有一个字段），测试通过后再加字段
